# F♭m code golf — compiled-byte scores

This folder holds programs aligned with classic [code.golf](https://code.golf) holes. For F♭m, the practical golf score is the size of the compiled program: **optimized `.ffb` bytes**. Source-file byte and character counts are still tracked for reference, and they follow the same definition as on code.golf ([About → “How are solutions scored?”](https://code.golf/about)):

- **Bytes:** length of the source file in UTF-8 octets.
- **Chars:** number of Unicode code points in the source (each code point counts as one stroke, regardless of UTF-8 width).

On code.golf, **leaderboard points** for a hole in a given language use the shortest solution in that language as the baseline: the best submission scores **1,000** points; a solution twice as long scores **500**, and so on (inverse proportion to length). The **global** leaderboard blends in a *par* derived from all languages; see the About page for the full formula.

For this repository's F♭m golf table, though, when we say a program's "score" we mean its **compiled byte size after optimization**, not its source strokes. That reflects the language design more directly: different surface syntaxes can compile to the same bytecode, so the interesting local comparison is the size of the generated `.ffb` payload.

## Scorecard

[`SCORECARD.md`](SCORECARD.md) is regenerated by [`score_golf.py`](score_golf.py). It lists every `.ff` / `.ffp` in this directory with:

- source **bytes** and **chars** (code.golf-style reference metrics),
- **optimized .ffb bytes**: the actual F♭m golf score — length of compiler stdout with the IR optimizer enabled, including the `FbAbbCb` header plus base64 bytecode (same layout as a `.ffb` file), produced by [`shell/ff-compile.sh`](../shell/ff-compile.sh) (wraps `deno/bin/ff-compile.ts --opt` and `shell/ff-preprocess.sh` for `.ffp`).

From the repository root:

```bash
python3 ff/golf/score_golf.py
```
